## Speed at Every Level of the Stack

> **Layer role:** Caching stores copies of frequently accessed data closer to where it's needed — in memory, at the edge, in the browser — to reduce latency and database load. A well-designed caching strategy can reduce API response times from 500ms to 5ms and eliminate 90%+ of database queries.

## Beginner Explanation

Imagine you work at a company with a huge library of documents in the basement. Every time someone asks you a question, you have to walk to the basement, find the document, walk back, and read the answer. This takes 5 minutes each time.

Now imagine keeping photocopies of the most-asked-about documents on your desk. Same question? Answer in 5 seconds. That's caching — keeping copies of data closer to where it's needed.

The challenge: when the original document changes (basement), your photocopy is outdated. This is the cache invalidation problem — one of the hardest problems in computer science.

## Cache Hierarchy

```
Fastest ↑                                                    Closest to user
─────────────────────────────────────────────────────────────────────────────
L1: Browser Memory Cache      ~0ms      JS variables, React state
L2: Browser HTTP Cache        ~0ms      Cached responses on disk
L3: Service Worker Cache      ~1ms      PWA offline cache
L4: CDN Edge Node            ~5-20ms    Closest geographic PoP
L5: API Server Memory        ~0.1ms    In-process object cache
L6: Redis / Memcached        ~1ms      Shared in-memory data store
L7: Read Replicas            ~5ms      DB read without write contention
L8: Primary Database         ~10-50ms  Source of truth
─────────────────────────────────────────────────────────────────────────────
Slowest ↓                                                   Furthest from user

Cache hit: Return from current level
Cache miss: Go to next level, then backfill current level
```

    B->>CDN: GET /index.html
    CDN-->>B: index.html + Cache-Control: no-cache, must-revalidate

### Cache-Control Headers

```
Cache-Control: max-age=31536000, immutable
  → Cache for 1 year. Content never changes (hashed filename).
  → Use for: JS bundles, CSS, images with content-hash filenames
  → app.abc123.js — filename changes when content changes

Cache-Control: no-cache
  → Must revalidate with server before using cached version.
  → Use for: index.html (must always have latest JS references)
  → With ETag: only downloads body if content changed (304)

Cache-Control: no-store
  → Never cache this. Never write to disk.
  → Use for: sensitive responses (banking, health data)
  → Strict privacy requirement

Cache-Control: public, max-age=300
  → Cache for 5 minutes, CDN and browser both cache.
  → Use for: news articles, product pages (change occasionally)

Cache-Control: private, max-age=3600
  → Cache only in browser (not CDN). User-specific data.
  → Use for: user profile, dashboard data
```

```nginx
# Nginx cache headers by file type
location ~* \.(js|css|woff2)$ {
    # Hashed filenames — immutable for 1 year
    add_header Cache-Control "public, max-age=31536000, immutable";
}

location ~* \.(png|jpg|webp|svg)$ {
    # Images — cache for 30 days (may change)
    add_header Cache-Control "public, max-age=2592000";
    add_header Vary "Accept";  # Vary by Accept header for WebP negotiation
}

location = /index.html {
    # HTML — revalidate every request
    add_header Cache-Control "no-cache, must-revalidate";
}

location /api/ {
    # API responses — no browser caching (CDN may cache via headers)
    add_header Cache-Control "no-store";
}
```

## CDN Architecture

A CDN (Content Delivery Network) has hundreds of servers (Points of Presence / PoPs) globally. When a user requests a resource, they're served from the nearest PoP.

**What CDNs cache:**
- Static assets (JS, CSS, images, fonts, videos)
- API responses (with appropriate Cache-Control)
- Rendered HTML (for SSR/ISR pages)

**What CDNs cannot cache:**
- User-specific responses (must have `private` or no-cache headers)
- POST/PUT/DELETE requests (by default)
- Responses with `Set-Cookie` headers (unless configured)

## Application Cache (Redis)

Redis is the workhorse of application-level caching. In-memory, ~1ms latency, rich data structures.

```typescript
import { createClient } from 'redis'

const redis = createClient({ url: process.env.REDIS_URL })
await redis.connect()

// ── Cache-Aside Pattern (most common) ───────────────────────
async function getUserProfile(userId: string): Promise<UserProfile> {
  const cacheKey = `user:${userId}:profile`

  // 1. Check cache
  const cached = await redis.get(cacheKey)
  if (cached) {
    return JSON.parse(cached)
  }

  // 2. Cache miss — fetch from DB
  const user = await db.user.findUnique({
    where: { id: userId },
    include: { profile: true, stats: true }
  })
  if (!user) throw new NotFoundError('User', userId)

  // 3. Store in cache with TTL
  const ttl = 300  // 5 minutes
  await redis.setex(cacheKey, ttl, JSON.stringify(user))

  return user
}

// ── Cache Invalidation ──────────────────────────────────────
async function updateUserProfile(userId: string, data: UpdateProfileInput) {
  // Update database first
  await db.user.update({ where: { id: userId }, data })

  // Invalidate cache — next read fetches fresh from DB
  await redis.del(`user:${userId}:profile`)

  // If user appears in lists/feeds, invalidate those too
  await redis.del(`feed:followers:${userId}`)
}

// ── Complex Cached Object (sorted set for leaderboard) ──────
async function updateLeaderboard(userId: string, score: number) {
  await redis.zadd('leaderboard:weekly', { score, member: userId })
}

async function getTopPlayers(limit = 10) {
  const cacheKey = `leaderboard:top:${limit}`
  const cached = await redis.get(cacheKey)
  if (cached) return JSON.parse(cached)

  // Fetch top N from sorted set (scores stored in Redis already)
  const topIds = await redis.zrevrange('leaderboard:weekly', 0, limit - 1, 'WITHSCORES')

  // Batch fetch user data
  const users = await db.user.findMany({
    where: { id: { in: topIds.filter((_, i) => i % 2 === 0) } }
  })

  const result = users.map((u, i) => ({ ...u, score: parseFloat(topIds[i * 2 + 1]) }))
  await redis.setex(cacheKey, 60, JSON.stringify(result))  // Cache 1 minute
  return result
}
```

### Redis Data Structures

| Structure | Example Key | Use Case | Command |
|-----------|------------|---------|---------|
| **String** | `user:123:profile` | Simple object cache | `GET/SET` |
| **Hash** | `session:abc123` | Session data (field-level access) | `HGET/HSET` |
| **List** | `notifications:user:123` | Activity feed, job queue | `LPUSH/LRANGE` |
| **Set** | `online:users` | Unique items, membership | `SADD/SMEMBERS` |
| **Sorted Set** | `leaderboard:2024` | Rankings, time-series | `ZADD/ZRANGE` |
| **Stream** | `events:purchases` | Event log, pub/sub | `XADD/XREAD` |
| **Bitmap** | `daily:active:2024-01` | Counting unique events efficiently | `SETBIT/BITCOUNT` |

## Cache Invalidation Strategies

> "There are only two hard things in Computer Science: cache invalidation and naming things." — Phil Karlton

### Strategy 1: TTL (Time to Live)

```typescript
// Simplest strategy: let cache expire on its own
await redis.setex('user:123', 300, JSON.stringify(user))  // Expires in 5 minutes

// Pros: Simple, no explicit invalidation needed
// Cons: Stale data for up to TTL duration

// Choose TTL based on:
// - How often data changes
// - How much staleness is acceptable
// - Database load savings

const ttls = {
  'user:profile': 300,       // 5 min — changes occasionally
  'post:123': 3600,          // 1 hour — rarely changes
  'global:config': 86400,    // 1 day — almost never changes
  'feed:latest': 60,         // 1 minute — changes frequently
  'session:*': 1800,         // 30 min — auto-expire idle sessions
  'trending:topics': 300,    // 5 min — aggregate, okay if slightly stale
}
```

### Strategy 2: Write-Through Cache

```typescript
// Write to cache AND database simultaneously on every update
async function updatePost(postId: string, data: UpdatePostInput) {
  const [updatedPost] = await Promise.all([
    db.post.update({ where: { id: postId }, data }),
    redis.setex(`post:${postId}`, 3600, JSON.stringify({
      ...existingPost,
      ...data,
      updatedAt: new Date().toISOString(),
    })),
  ])

  return updatedPost
}

// Pros: Cache always consistent with DB
// Cons: Every write touches both DB and cache (higher write latency)
// Best for: Read-heavy with infrequent writes
```

### Strategy 3: Cache-Aside (Lazy Loading)

```typescript
// Only cache on read misses. Explicitly invalidate on writes.
// Most common pattern — already shown above

// Pros: Only cache data that's actually read, simple
// Cons: Cache cold on startup, first request after invalidation is slow
```

### Strategy 4: Event-Driven Invalidation

```typescript
// Database trigger or application event invalidates cache
// Best for: complex invalidation logic, multiple cache keys per data change

// Subscribe to database changes via CDC (Change Data Capture)
// PostgreSQL LISTEN/NOTIFY
await db.query(`LISTEN user_updates`)
db.on('notification', async (msg) => {
  const { userId } = JSON.parse(msg.payload)
  // Invalidate all cache keys related to this user
  const keys = await redis.keys(`user:${userId}:*`)
  if (keys.length) await redis.del(keys)
})

// Application event (when you control the write)
eventBus.on('user.profile.updated', async ({ userId }) => {
  await Promise.all([
    redis.del(`user:${userId}:profile`),
    redis.del(`user:${userId}:feed`),
    redis.del(`suggestions:*`),  // User profile affects suggestions
  ])
})
```

## Cache Patterns

### Cache Stampede (Thundering Herd) Problem

```
Problem:
  1. Cache key expires (TTL hit)
  2. 1000 concurrent requests check cache — all miss simultaneously
  3. All 1000 requests query the database
  4. Database overwhelmed with 1000 identical queries
  5. System either slows dramatically or crashes

Solutions:
```

```typescript
// Solution 1: Lock (mutex) — only one request queries DB
async function getCachedUser(userId: string) {
  const cacheKey = `user:${userId}`
  const lockKey = `lock:${cacheKey}`

  // Try cache first
  const cached = await redis.get(cacheKey)
  if (cached) return JSON.parse(cached)

  // Try to acquire lock
  const locked = await redis.set(lockKey, '1', { NX: true, EX: 5 })

  if (locked) {
    try {
      const user = await db.user.findUnique({ where: { id: userId } })
      await redis.setex(cacheKey, 300, JSON.stringify(user))
      return user
    } finally {
      await redis.del(lockKey)
    }
  } else {
    // Another request has the lock — wait briefly and re-check cache
    await new Promise(resolve => setTimeout(resolve, 100))
    const retryCache = await redis.get(cacheKey)
    if (retryCache) return JSON.parse(retryCache)
    // Fall through to DB (lock may have failed/expired)
    return db.user.findUnique({ where: { id: userId } })
  }
}

// Solution 2: Early expiration + background refresh
async function getCachedUserEarlyExpire(userId: string) {
  const cached = await redis.get(`user:${userId}`)
  if (cached) {
    const { data, expiresAt } = JSON.parse(cached)
    // If expiring within next 30s, refresh in background
    if (expiresAt - Date.now() < 30000) {
      refreshInBackground(userId)  // Don't await — update quietly
    }
    return data
  }
  // ... cache miss logic
}
```

// Prisma extension for automatic caching

## Common Mistakes

### 1. Caching User-Specific Data in CDN
```typescript
// ❌ User's dashboard page cached at CDN — User A sees User B's data!
res.setHeader('Cache-Control', 'public, max-age=3600')

// ✅ Private (per-user) responses are browser-only
// User-specific API responses — no CDN caching
res.setHeader('Cache-Control', 'private, no-cache')

// ✅ Use Vary header if response varies by cookie/header
res.setHeader('Vary', 'Cookie, Accept-Language')
```

### 2. Caching Without Considering Consistency
```typescript
// ❌ 1-hour cache on user balance — user pays, sees stale balance
await redis.setex(`user:${id}:balance`, 3600, balance.toString())

// ✅ Short TTL or invalidate on write
// For financial data: either very short TTL or no cache at all
// If you must cache: invalidate immediately on balance change
async function recordPayment(userId: string, amount: number) {
  await db.account.update({ ... })
  await redis.del(`user:${userId}:balance`)  // Immediate invalidation
}
```

### 3. No Cache Warm-up
```bash
# ❌ After deploy: all caches empty → DB hammered → site slow for 10 min
# ✅ Warm up cache before switching traffic (blue-green)
# Pre-populate top N most-requested items before deploying
node scripts/warm-cache.js --top-users=10000 --top-posts=50000
```

## Best Practices

1. **Cache at the right level** — Static assets → CDN. Computed objects → Redis. User session → Redis. DB queries → read replica or Redis.
2. **Set appropriate TTLs** — Don't over-cache changing data. Don't under-cache stable data.
3. **Cache keys must be unique** — Include all dimensions: `user:{userId}:posts:{page}:{sort}`
4. **Monitor hit rates** — A hit rate below 70% means your caching strategy needs work.
5. **Plan for stampedes** — Use locks or probabilistic early refresh for popular keys.

## Interview-Level Insights

### Q: What is the difference between write-through and write-back cache?

**A:** Write-through: on every write, update both cache and database synchronously. Cache is always consistent with DB. Higher write latency. Safer — no data loss if cache crashes.

Write-back (write-behind): write to cache first, return success, then asynchronously write to database. Lower write latency, risk of data loss if cache crashes before the background write completes. Used in storage systems (SSDs use write-back internally).

For web applications: write-through or cache-aside are most common. Write-back is too risky for most OLTP workloads.

### Q: How does a CDN cache work, and how do you invalidate it?

**A:** CDN edge nodes cache responses based on the URL + Vary headers + Cache-Control headers from the origin. When a cache miss occurs, the edge node fetches from origin and stores the response.

Invalidation: 1) Wait for TTL expiration (simplest), 2) Programmatic purge via CDN API (Cloudflare API, CloudFront invalidations), 3) URL versioning (cache-busting via content hash in filename — no invalidation needed).

URL versioning is the best practice for static assets — change the filename when content changes. No invalidation needed, perfect cache hits.

## Summary

Caching is one of the highest-leverage optimizations in system design:

1. **Browser cache** — Immutable content-hashed filenames, no-cache for HTML
2. **CDN** — Edge-served static assets globally, millisecond latency
3. **Redis** — Shared application cache, sub-millisecond access, rich data structures
4. **Cache-aside** — The most flexible and common pattern
5. **Invalidation** — TTL for simplicity, event-driven for consistency

A 90% cache hit rate on your API means 90% fewer database queries. At scale, this is the difference between a $10K database and a $100K one.

## Production Checklist

- [ ] Static assets served with content-hash filenames (immutable caching)
- [ ] CDN configured for static assets
- [ ] Cache-Control headers set correctly per resource type
- [ ] Redis cluster (not single node) for HA
- [ ] Redis persistence configured (AOF or RDB) for durability
- [ ] Cache key naming convention documented
- [ ] TTL values chosen based on data change frequency
- [ ] Cache invalidation logic for all write paths
- [ ] Cache hit rate monitored (target > 80%)
- [ ] Stampede protection on high-traffic keys
- [ ] No user-specific data in CDN cache (Vary header or private)
- [ ] Redis maxmemory policy set (allkeys-lru for cache use case)
- [ ] Redis memory usage monitored and alerting configured
