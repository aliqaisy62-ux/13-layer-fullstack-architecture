## Where Your Application Lives and How It Gets There

> **Layer role:** Hosting is the infrastructure that runs your application. Deployment is the process of getting new code to that infrastructure. Getting these right means your app is reliable, easy to update, and can recover from failures. Getting them wrong means weekend oncall calls and production fires.

## Beginner Explanation

Hosting is simply where your app runs. Your laptop can run code locally, but users can't access your laptop. Hosting means putting your code on a computer (a server) that is:
- Always on (24/7 uptime)
- Connected to the internet with a public IP address
- Powerful enough to handle many simultaneous users

Deployment is how you get new code onto those servers. Early teams do it manually via SSH. Professional teams automate it with pipelines that test, build, and deploy every code change.

## Containers & Docker

Docker packages your application and all its dependencies into a **container** — a consistent, isolated environment that runs identically everywhere.

```
"It works on my machine" → Solved by Docker
Your machine: Node 20, Ubuntu 22.04, specific npm packages
Their server:  Node 18, CentOS 7, different packages  ← Bug

Docker container: Node 20, exact npm packages, baked in
Same everywhere: dev laptop, CI runner, staging, production ← No bug
```

### Dockerfile

```dockerfile
# ─── Build stage ─────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

# Copy package files first (cache optimization)
# If these don't change, Docker reuses the cached layer
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force

# Copy source and build
COPY . .
RUN npm run build

# ─── Production stage ────────────────────────────────────────
# Start fresh — only include what's needed to run
FROM node:20-alpine AS production

# Security: run as non-root user
RUN addgroup -S app && adduser -S app -G app
WORKDIR /app

# Copy only built files and production dependencies
COPY --from=builder --chown=app:app /app/dist ./dist
COPY --from=builder --chown=app:app /app/node_modules ./node_modules
COPY --from=builder --chown=app:app /app/package.json .

USER app

EXPOSE 3000

# Health check — Kubernetes/load balancer uses this
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "dist/main.js"]
```

## Deployment Strategies

Choosing the right deployment strategy determines how much risk each release carries.

### Blue-Green Deployment

```mermaid
graph TB
    subgraph Before["Before Deploy"]
        LB1[Load Balancer] -->|100% traffic| BLUE[Blue Environment\nv1.0 - Current]
        GREEN1[Green Environment\nv2.0 - New] --- IDLE[Idle / Warming up]
    end

    subgraph Switch["Instant Switch"]
        LB2[Load Balancer] -->|0% traffic| BLUE2[Blue v1.0]
        LB2 -->|100% traffic| GREEN2[Green v2.0]
        Note1[Blue kept running\nfor instant rollback]
    end

    style Before fill:#dbeafe
    style Switch fill:#dcfce7
```

**Blue-Green process:**
1. Deploy new version to "green" environment
2. Run smoke tests against green
3. Switch load balancer to send all traffic to green (instant cutover)
4. Keep blue running for 15-30 minutes (instant rollback if problems)
5. Decommission blue after confidence period

**Best for:** Minimal downtime, instant rollback, high-risk releases.

---

### Canary Deployment

```mermaid
graph LR
    LB[Load Balancer] -->|95% traffic| OLD[v1.0 - Stable]
    LB -->|5% traffic| NEW[v2.0 - Canary]

    NEW --> METRICS[Monitor Error Rate\nLatency · Conversions]
    METRICS -->|Metrics OK| ROLLOUT[Gradually increase %\n5% → 25% → 50% → 100%]
    METRICS -->|Metrics bad| ROLLBACK[Rollback to 0%]
```

**Canary process:**
1. Deploy new version alongside current
2. Route 5% of users to new version
3. Monitor: error rate, latency, business metrics
4. Gradually increase percentage if metrics are healthy
5. Roll back if anything looks wrong

**Best for:** High-traffic systems, risk-averse teams, A/B testing new features.

---

### Rolling Deployment

```
Servers: [v1] [v1] [v1] [v1] [v1] [v1]

Step 1:  [v2] [v1] [v1] [v1] [v1] [v1]  ← Replace one at a time
Step 2:  [v2] [v2] [v1] [v1] [v1] [v1]
Step 3:  [v2] [v2] [v2] [v1] [v1] [v1]
...
Final:   [v2] [v2] [v2] [v2] [v2] [v2]
```

**Best for:** Kubernetes deployments (default strategy), gradual rollouts.

**Risk:** During rollout, v1 and v2 run simultaneously. Your API must be backward-compatible (see "Backwards Compatibility" in best practices).

---

### Feature Flags

The most mature pattern: deploy code to 100% of servers but gate features with flags. Decouple deployment from release.

```typescript
// LaunchDarkly / GrowthBook / Unleash
import { launchDarkly } from './featureFlags'

async function getCheckoutFlow(userId: string) {
  const useNewCheckout = await launchDarkly.variation('new-checkout-v2', {
    key: userId,
    // Can target by: userId, email, plan, country, random %
  }, false)  // default = false

  if (useNewCheckout) {
    return renderNewCheckout()
  }
  return renderOldCheckout()
}

// Benefits:
// - Deploy code to all servers at once
// - Enable for 1% of users, monitor, then ramp up
// - Instant kill switch: disable flag, feature off globally in seconds
// - No new deployment needed to turn feature on/off
```

import { z } from 'zod'

  DATABASE_URL: z.string().url(),

## Serverless Deployment

Serverless means you deploy functions, not servers. The cloud provider runs them on demand.

```mermaid
graph LR
    REQ[HTTP Request] --> EDGE[Edge Network]
    EDGE --> COLD{Function\nCold?}
    COLD -- Yes --> INIT[Initialize Container\n~200ms cold start]
    COLD -- No --> WARM[Reuse Container\n~1ms warm start]
    INIT --> EXEC[Execute Function\n~50ms]
    WARM --> EXEC
    EXEC --> RES[Response]

    style COLD fill:#fef9c3
    style INIT fill:#fee2e2
    style WARM fill:#dcfce7
```

```typescript
// Vercel Function (serverless)
// File: pages/api/users/[id].ts
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(req: NextApiRequest, res: NextApiResponse) {
  const { id } = req.query

  if (req.method === 'GET') {
    const user = await db.user.findUnique({ where: { id: id as string } })
    if (!user) return res.status(404).json({ error: 'Not found' })
    return res.json(user)
  }

  if (req.method === 'PUT') {
    // ...
  }

  res.setHeader('Allow', ['GET', 'PUT'])
  res.status(405).end('Method Not Allowed')
}

export const config = {
  runtime: 'edge',  // Run at CDN edge — ultra-low latency globally
}
```

**Serverless pros:**
- Zero server management
- Auto-scales to zero (no traffic = no cost)
- Instant global deployment
- Pay per invocation

**Serverless cons:**
- Cold starts (first request after idle can be slow)
- Execution time limits (15min max for Lambda)
- Not suitable for: WebSockets, long-running processes, stateful apps
- Database connections are problematic (use connection poolers like PgBouncer, Neon, Supabase)

## Common Mistakes

### 1. Same Config Across Environments
```yaml
# ❌ Using production DB URL in staging — staging tests corrupt prod data
DATABASE_URL: postgresql://prod-host/myapp

# ✅ Separate environments, separate resources
staging:
  DATABASE_URL: postgresql://staging-host/myapp_staging
production:
  DATABASE_URL: postgresql://prod-host/myapp
```

### 2. Not Testing Rollback
```bash
# ❌ Only testing forward deployments
# When rollback is needed (3am production incident), nobody knows how

# ✅ Practice rollbacks in staging monthly
# Document rollback procedures
# Kubernetes rollback:
kubectl rollout undo deployment/api --to-revision=5
# Verify rollback worked:
kubectl rollout status deployment/api
```

### 3. Missing Health Checks
```typescript
// ❌ Load balancer routes traffic to crashed container

// ✅ Health endpoint checks all dependencies
app.get('/health', async (req, res) => {
  const checks = await Promise.allSettled([
    db.query('SELECT 1'),
    redis.ping(),
  ])

  const healthy = checks.every(c => c.status === 'fulfilled')
  res.status(healthy ? 200 : 503).json({
    status: healthy ? 'healthy' : 'degraded',
    checks: {
      database: checks[0].status,
      redis: checks[1].status,
    }
  })
})
```

## Best Practices

1. **Immutable deployments** — Never modify running containers. Deploy new containers, shift traffic, kill old ones.
2. **Zero-downtime deploys** — Use rolling/blue-green. Maintenance windows are a last resort.
3. **Backwards-compatible changes** — When adding a column, add it nullable first. Never rename a column in one deploy.
4. **Health checks everywhere** — Load balancer, Kubernetes, monitoring system all check `/health`.
5. **Automate everything** — If you're SSHing to deploy, you will forget a step at 2am.

## Interview-Level Insights

### Q: Explain the 12-factor app methodology.

**A:** The 12-factor app is a methodology for building software-as-a-service apps that are: portable across environments, cloud-native, and can scale. Key factors relevant to hosting:

- **Codebase**: One codebase, multiple deploys
- **Config**: Store config in environment variables (not in code)
- **Processes**: Run as stateless processes (state in DB/Redis, not in-memory)
- **Port binding**: Export services via port binding (self-contained HTTP server)
- **Concurrency**: Scale out via the process model (horizontal, not vertical)
- **Disposability**: Fast startup, graceful shutdown — containers are cattle, not pets

### Q: What's the difference between a blue-green and a canary deployment?

**A:** Blue-green is binary — 100% of traffic switches instantly. It's safe (easy rollback), but provides no gradual validation. Canary is gradual — start with 5%, grow to 100% based on metrics. It validates new code with real users before full rollout. Blue-green is simpler; canary is safer for large user bases.

## Summary

Hosting and deployment are where code meets reality. The key principles:

1. **Containerize everything** — Docker makes environments reproducible
2. **Automate deployments** — Human error causes most outages
3. **Choose a strategy** — Rolling for low risk, canary for high-traffic, blue-green for instant rollback
4. **Separate environments** — Dev, staging, production must be isolated
5. **Health check everything** — Know before users do

---

## Production Checklist

- [ ] Docker multi-stage builds (minimal production image)
- [ ] Non-root user in containers
- [ ] Health check endpoint on every service
- [ ] Zero-downtime deployment strategy chosen
- [ ] Rollback procedure documented and tested
- [ ] Environment variables validated at startup
- [ ] Secrets in secrets manager (not .env files)
- [ ] Staging environment mirrors production
- [ ] Container registry with image scanning enabled
- [ ] Resource limits set on containers (CPU/memory)
- [ ] Graceful shutdown handler (SIGTERM)
- [ ] Feature flags for risky changes
